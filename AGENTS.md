# Repository Guidelines

## Project Structure & Module Organization

`./src/index.ts` is the only entry point exposed by the package and curates the public surface; nothing under `./src/internal/` is exported. The library is split by concern:

- `./src/generator.ts` defines the `Generator<T>` abstract base (the monad: `map`, `flatMap`, `pure`, `sample`, `sampleN`, `withShrinker`) plus `GenError` and `SampleOpts`.
- `./src/generators/` holds the built-in combinators (`array`, `boolean`, `collection`, `core`, `date`, `filter`, `ntuple`, `number`, `object`, `option`, `pick`, `sized`, `string`); `./src/generators/index.ts` re-exports them as the `Generators` namespace.
- `./src/quantifiers/` contains `forAll`, `forAll.assert`, `exists`, and the shrinking driver.
- `./src/random/` provides the deterministic RNG: `mulberry32` advanced by a `cyrb53`-hashed seed (`number | string`).
- `./src/maybe.ts` and `./src/lazy.ts` back the empty-case (`none`) and `Generators.recursive` thunking.

Tests live in `./spec/` and mirror the source layout — per-combinator suites under `./spec/generators/`. Runnable examples in `./examples/` double as integration tests and are executed by `./scripts/run-examples.mjs`. VitePress sources live under `./docs/`, with `./docs/api/` generated by TypeDoc.

## Build, Test, and Development Commands

- `npm test` — run the Vitest suite once.
- `npx vitest run spec/generators/number.test.ts` — run a single test file.
- `npm run lint` / `npm run typecheck` — ESLint and `tsc --noEmit` against both `./tsconfig.json` and `./tsconfig.test.json`.
- `npm run test:examples` — execute every `./examples/*.ts` via `tsx`.
- `npm run build` — produce dual ESM/CJS bundles plus `.d.ts` into `./dist/` via tsup.
- `npm run docs:dev` / `npm run docs:build` — regenerate TypeDoc API pages and run/build the VitePress site.
- `npm run check` — the full gate CI uses: lint + typecheck + tests + examples + `docs:check`.

## Coding Style & Naming Conventions

Prettier (`./.prettierrc`): 2-space indent, single quotes, semicolons, trailing commas everywhere, 100-column width, always-parenthesised arrow params. ESLint extends `typescript-eslint/recommended`; `no-explicit-any` and `prefer-const` are disabled, unused `_`-prefixed identifiers are allowed. The `tsdoc/syntax` rule runs only on `./src/**/*.ts` — keep public API doc comments TSDoc-valid. TypeScript runs in strict mode with `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, and `verbatimModuleSyntax`, so use `import type` for type-only imports and relative imports must include the `.js` extension. ESLint ignores `./examples/`.

## Testing Guidelines

Vitest (`./vitest.config.ts`) picks up `spec/**/*.test.ts` in the Node environment with `globals: false` — import `describe`, `it`, `expect` from `vitest` explicitly. Name files `<module>.test.ts` mirroring the source under `./src/`. New combinators need both a unit suite under `./spec/generators/` and, where user-facing, a runnable script under `./examples/` (it will be picked up automatically by `test:examples`).

## Commit & Pull Request Guidelines

Commits use short imperative sentences with a capitalised first word and no Conventional Commits prefix — e.g. `Implement "recursive"`, `Add tests for nonEmptyArray`, `Fix a bug in the oneOf generator combinator code`. Refactors are tagged in-message (`Re-factor ...`, `Minor re-factorings`). There is no PR template; before opening a PR run `npm run check` locally — the `./.github/workflows/ci.yml` matrix runs it on Node 22 and 24 and then `npm run build`.
