# Contributing to check-loki

Thanks for taking the time to hack on **check-loki**. This guide describes how to
build, test, and try the npm package locally.

## Prerequisites

- **Node.js >= 22** (the `engines` field in `./package.json` enforces this; CI runs on 22 and 24).
- **npm** (ships with Node).
- A POSIX-like shell. Windows users should use WSL or Git Bash for the helper scripts in `./scripts/`.

Check your toolchain:

```sh
node --version   # should print v22.x or newer
npm --version
```

## Getting the source

```sh
git clone git@github.com:amoilanen/check-loki.git
cd check-loki
npm install
```

`npm install` installs the dev dependencies declared in `./package.json` — TypeScript, ESLint, Prettier, Vitest, tsup, TypeDoc, and VitePress. There are **no runtime dependencies**.

## Project layout

A quick map so you know where to put things (see `./AGENTS.md` for the full version):

- `./src/` — library source. `./src/index.ts` is the only public entry point.
- `./src/generators/` — built-in combinators re-exported as the `Generators` namespace.
- `./spec/` — Vitest unit tests, mirroring the layout of `./src/`.
- `./examples/` — runnable `.ts` scripts that double as integration tests.
- `./scripts/run-examples.mjs` — driver that executes every example via `tsx`.
- `./docs/` — VitePress site sources; `./docs/api/` is generated by TypeDoc.
- `./dist/` — build output (gitignored, produced by `npm run build`).

## Build the package

The package is built with [tsup](https://tsup.egoist.dev/), configured in `./tsup.config.ts`. It produces dual ESM + CJS bundles and `.d.ts` type declarations.

```sh
npm run build
```

Output lands in `./dist/`:

- `./dist/index.mjs` — ESM entry (`"module"` / `"exports".import`)
- `./dist/index.cjs` — CommonJS entry (`"main"` / `"exports".require`)
- `./dist/index.d.ts` — TypeScript declarations

Only the `./dist/` directory is shipped to npm (see the `"files"` field in `./package.json`).

## Run the tests

### Unit tests

```sh
npm test                                         # full Vitest suite, single run
npx vitest                                       # watch mode
npx vitest run spec/generators/number.test.ts    # a single file
```

Vitest is configured in `./vitest.config.ts`. Tests run in the Node environment with `globals: false`, so import `describe`, `it`, and `expect` from `vitest` explicitly. New test files belong under `./spec/` and should be named `<module>.test.ts`, mirroring the source under `./src/`.

### Example scripts

Every `.ts` file under `./examples/` is executed end-to-end by:

```sh
npm run test:examples
```

The driver lives at `./scripts/run-examples.mjs` and runs each example through `tsx`. Adding a new file to `./examples/` automatically opts it into this check.

### Lint and type-check

```sh
npm run lint        # ESLint on the whole repo (config: ./eslint.config.js)
npm run typecheck   # tsc --noEmit against ./tsconfig.json and ./tsconfig.test.json
```

Formatting follows `./.prettierrc` (2-space indent, single quotes, semicolons, trailing commas, 100-column width).

### The full gate

CI (`./.github/workflows/ci.yml`) runs the aggregate `check` script. Run it locally before opening a PR:

```sh
npm run check
```

This runs, in order: `lint` → `typecheck` → `test` → `test:examples` → `docs:check`.

## Documentation

The site sources live under `./docs/` and are built with VitePress; the API reference under `./docs/api/` is generated from TSDoc comments by TypeDoc.

```sh
npm run docs:api      # regenerate ./docs/api/ from TSDoc
npm run docs:dev      # local dev server with hot reload
npm run docs:build    # production build into ./docs/.vitepress/dist
npm run docs:preview  # serve the production build
npm run docs:check    # TypeDoc in --emit none mode; part of `npm run check`
```

When you add or change a public symbol in `./src/`, keep its TSDoc comment valid — `docs:check` will fail otherwise, and the `tsdoc/syntax` ESLint rule runs on `./src/**/*.ts`.

## Trying the package in another project

Sometimes you want to consume your local build from a separate project before publishing. Two common workflows:

### Option 1: `npm pack` (recommended, closest to a real install)

From the check-loki repo:

```sh
npm run build
npm pack          # produces check-loki-<version>.tgz
```

Then in the consumer project:

```sh
npm install /absolute/path/to/check-loki-<version>.tgz
```

This installs exactly what would be published — only the files listed under `"files"` in `./package.json` (i.e. `./dist/`) end up in `node_modules`.

### Option 2: `npm link`

From the check-loki repo:

```sh
npm run build
npm link
```

Then in the consumer project:

```sh
npm link check-loki
```

Re-run `npm run build` (or use `npx tsup --watch`) after each source change so the linked `./dist/` stays up to date. Unlink afterwards with `npm unlink check-loki` in the consumer and `npm unlink -g check-loki` in this repo.

## Commit and PR conventions

- Use **short imperative sentences** with a capitalised first word and **no Conventional Commits prefix**. Examples from the history: `Implement "recursive"`, `Add tests for nonEmptyArray`, `Fix a bug in the oneOf generator combinator code`.
- Tag refactors in-message (`Re-factor ...`, `Minor re-factorings`).
- Run `npm run check` locally before opening a PR — CI runs the same gate on Node 22 and 24, then `npm run build`.
